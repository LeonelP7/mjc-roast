.What is spring data?
[%collapsible]
====
Is a spring project that aims to simplify database access by reducing the amount of boilerplate code you have to write. Allowing you to focus on your business logic instead.

It sits on top of Hibernate and JPA to provide a consistent, interface-based way to interact with your data.

Its most famous feature is the ability to create Repository interfaces by simply extending `JpaRepository`, Spring automatically generates all the CRUD implementation code at runtime.

It manages pagination and sorting out of the box

One of the most impressive "magic" tricks of Spring Data is Query Methods. Instead of writing SQL or JPQL, you can simply define a method name in your interface like findByLastNameAndAge(String name, int age), and Spring Data will parse the method name and automatically generate the correct query logic for you.

It isn't limited to just relational databases either; the "Spring Data" umbrella covers many modules, including Spring Data MongoDB, Spring Data Redis, and Spring Data Cassandra, providing a familiar and consistent programming model regardless of whether you are using a SQL or NoSQL database.

.JDBC → Hibernate → JPA → Spring Data
[%collapsible]
====
1. JDBC (The Manual Foundation)
+
In the beginning, there was only JDBC. It was revolutionary because it allowed Java to talk to any database, but it was incredibly tedious.
+
*The Problem:* You had to write hundreds of lines of "boilerplate" code—manually opening connections, handling SQL strings as text, and mapping every single database column to a Java field by hand.
+
*The Vibe:* Like building a car from scratch using raw metal and rubber.

2. Hibernate (The Automated Engine)
+
Developers got tired of mapping columns to fields, so Hibernate was born as an ORM (Object-Relational Mapping) tool.
+
*The Solution:* It allowed you to simply "map" a Java class to a database table. You stopped writing `INSERT INTO...` and started calling `session.save(user)`.
+
*The Innovation:* It introduced features like the First-Level Cache and Lazy Loading to make database access faster without extra effort.
+
*The Vibe:* Like buying a pre-assembled engine so you don't have to build the cylinders yourself.

3. JPA (The Universal Standard)
+
As more ORM tools appeared (like Hibernate, TopLink, and iBatis), the industry realized we needed a Standard. If you wrote code for Hibernate and wanted to switch to another tool, you had to rewrite everything.
+
*The Solution:* JPA was created as a "specification" (a set of rules). It provided standard annotations like `@Entity` and `@Table`.
+
*The Innovation:* Hibernate changed its code to follow these rules. Now, your code is "Standard Java," and Hibernate is just the engine running under the hood.
+
*The Vibe:* Like creating a universal driver's license that works in any car, regardless of the brand.

4. Spring Data (The Ultimate Abstraction)
+
Even with JPA and Hibernate, you still had to write "DAO" (Data Access Object) classes and repeat the same find, save, and delete logic for every single entity.
+
*The Solution:* Spring Data sits on top of everything. It says, "If you give me an interface, I will write the implementation for you."
+
*The Innovation:* It introduced Repositories. You define a method name like `findByEmail`, and Spring Data literally writes the SQL, executes it through JPA/Hibernate, and returns the result via JDBC.
+
*The Vibe:* Like a self-driving car—you just tell it where to go, and it handles the engine, the transmission, and the road.

.What is DTO and DAO?
[%collapsible]
====
DAO (Data Access Object): Is a pattern used to isolate the business layer from the persistence layer (the db). Its job is to handle all the work of database communication: Executing SQL queries, calling the EntityManager, or using JDBC.

DTO (Data Transfer Object): Is a simple object used to carry data between processes, usually between the Backend (Controller) and the Frontend (Client). Unlike an Entity (that matches your database table) a DTO only contains the specific fields needed for a particular task.

This reduce the number of remote calls (by grouping data) and to hide sensitive database structures.

.What are architecture Patterns?
[%collapsible]
====
Are strategies that provide reusable solutions to common problems in software design. While Design Pattern (like Singleton or Factory) focuses on how a few classes interact, an Architecure Pattern defines the fundamental structural organization of an entire system, by determining how major components relate and how data flows between them

1. Layered (N-Tier) Architecture
+
This is the most common pattern, especially for beginners and standard web applications. It organizes code into horizontal layers (Presentation, Business, Persistence, and Database).
+
The Goal: Separation of concerns. Each layer only speaks to the one immediately below it.
+
Best for: Small to medium-sized applications where you want a simple, organized structure.

2. Microservices Architecture
+
In this pattern, the application is split into small, independent services that communicate via APIs (like REST). Each service handles one specific business function (e.g., "Shipping Service," "Inventory Service").
+
The Goal: Scalability and agility. You can update or restart the "Payment Service" without affecting the rest of the app.
+
Best for: Large, complex enterprise systems with multiple development teams.

3. Event-Driven Architecture
+
This pattern relies on a central "Event Bus" or "Message Broker" (like Kafka). Instead of components calling each other directly, they "publish" events (e.g., "OrderPlaced") and other components "subscribe" to them.
+
The Goal: Asynchronous processing and high decoupling. The "Order" component doesn't need to wait for the "Email" component to finish sending a confirmation.
+
Best for: Real-time data processing and highly reactive systems.

4. Hexagonal (Ports and Adapters) Architecture
+
This pattern puts the business logic in the center (the "Core") and surrounds it with "Ports." Outside tools like Databases, APIs, or UIs connect via "Adapters."
+
The Goal: Technology independence. You can swap your database from MySQL to MongoDB just by changing an Adapter, without ever touching your core business logic.
+
Best for: Long-term projects where you want to keep your logic safe from changing tech trends.

5. Microkernel (Plugin) Architecture
+
This consists of a "Core System" that provides minimal functionality and "Plug-in Modules" that add specialized features. Think of your Web Browser or an IDE like VS Code.
+
The Goal: Extensibility. You can add new features without modifying the core engine.
+
Best for: Desktop applications or tools that require heavy customization.

.What are ingeenering principles?
[%collapsible]
====
Are a set of fundamental rules and best practices that guide developers in creating software that is reliable, easy to maintain, and scalable. Arele like philosophies that keep your code from becoming messy or fragile over time

1. SOLID Principles
+
This is the most famous set of principles in Object-Oriented Programming. It consists of five rules designed to make software designs more understandable and flexible:
+
S (Single Responsibility): A class should have only one reason to change (it should do one thing well).
+
O (Open/Closed): Code should be open for extension but closed for modification.
+
L (Liskov Substitution): You should be able to replace a parent class with a child class without breaking the app.
+
I (Interface Segregation): Don't force a class to implement methods it doesn't use. Instead having a bunch of methods in a single interfaces that all the classes that extend from it might not need them all, break the interface into smaller ones
+
D (Dependency Inversion): Depend on abstractions (interfaces), not concrete implementations.

2. DRY (Don't Repeat Yourself)
+
The DRY principle states that every piece of knowledge or logic must have a single, unambiguous representation within a system. If you find yourself copying and pasting the same code in three different places, you should move that logic into a single shared method or utility class.
+
The Goal: Reducing bugs. If you need to change a rule, you only have to change it in one place.

3. KISS (Keep It Simple, Stupid)
+
This principle emphasizes that systems work best if they are kept simple rather than made complicated. Developers often fall into the trap of "over-engineering"—writing complex solutions for simple problems just in case they might need the complexity later.
+
The Goal: Readability. Simple code is easier to debug and easier for a new teammate to understand.

4. YAGNI (You Ain't Gonna Need It)
+
Related to KISS, YAGNI suggests that you should never add functionality until it is actually necessary. Don't build a massive "Notification System" if you only need to send one email today.
+
The Goal: Focus. It prevents you from wasting time on features that might never be used.

5. Separation of Concerns (SoC)
+
This is the principle that leads to Layered Architecture. It dictates that different parts of your program should handle different responsibilities. For example, your database code shouldn't care how the user's browser looks, and your UI code shouldn't know how to save data.

.What is rabbitMQ?
[%collapsible]
====
Is an open-source message broker that acts as a "middleman" for communication between different applications or services. It uses a protocol called AMQP (Advanced Message Queuing Protocol) to accept messages from a producer, store them in a queque, and then deliver them to a consumer when its ready to process them.

Because RabbitMQ handles the logic of where messages go (using Exchanges) and ensures they aren't lost if a service crashes (using Persistence), it is an essential tool for building reliable, scalable systems. It is particularly popular in Microservices architectures and Event-Driven designs because it handles high volumes of messages and ensures that even if one part of your system is slow or offline, the rest of the application can keep functioning smoothly.
====

.What is the difference between an API and a message broker?
[%collapsible]
====
The difference is primary how the communication happens

API: Specifically a REST API, represents a direct connection, between two systems. When System A calls System B's API, it waits for a response before moving on. (Syncronous)

Message Broker: Acts as a intermediary. System A (the Producer) sends a message to the broker and then immediately goes back to its own work. The broker stores the message in a Queue until System B (the Consumer) is ready to pick it up. (Asynchronous)

[cols="1,2,2", options="header"]
|===
| Feature | API (REST/HTTP) | Message Broker (RabbitMQ/Kafka)
| Timing
| Synchronous: Wait for response.
| Asynchronous: Fire and forget.
| Availability
| Both must be online simultaneously.
| Producer can work even if Consumer is down.
| Data Flow
| Point-to-point (Direct).
| Through a Queue (Indirect).
| Standard Use
| Getting data (e.g., "Give me user profile").
| Triggering tasks (e.g., "Send this email").
|===

.What is Inversion of Control (IoC)?
[%collapsible]
====
Is a software engineering principle, is the philosophy of shifting the responsability to handle dependencies from the developer to the system.

It handles:

Object Creation: Instead of you writing new UserService(), the framework creates the instance.

Lifecycle Management: The framework decides when an object is born (instantiated) and when it dies (garbage collected).

Flow of Execution: In a console app, you write main() and call every method. In a Spring Web app, you just write a method with @GetMapping, and Spring "finds" and runs it when a user visits that URL.

.The IoC container
[%collapsible]
====
In Spring, the "Studio" that manages everything is called the IoC Container. It is the environment that:

Scans your code for components (using @Component, @Service, etc.).

Instantiates them.

Wires them together (Dependency Injection).

Manages their entire life inside the application.
====

.What is dependency injection?
[%collapsible]
====
Is a design pattern used to implement Inversion of Control (IoC). In simple term it is a technique where an object does not create the other object it relies on (its dependencies). Instead those dependencies are "injected" into it.

Three Ways to Inject

In Spring, there are three primary ways to perform this injection:

Constructor Injection (Best Practice): Dependencies are provided through the class constructor. This ensures the object is never created in an "incomplete" state.

Setter Injection: Dependencies are provided through setter methods after the object is created.

Field Injection: Dependencies are injected directly into fields (usually via the @Autowired annotation). While easy to write, this is generally discouraged because it makes testing harder.
====

.What is @Autowired?
[%collapsible]
====
Is a Spring annotation, used to perfrom Dependency Injection automatically. It tells Spring IoC (Inversion of Control) Container: "Find a bean that matches this data type and plug it in here for me"

When your Spring application starts up, it performs a "component scan" to find all the classes marked with annotations like @Service, @Repository, or @Component. Once it has created these objects (called Beans), it looks for the @Autowired annotation to see which beans need to be injected into others.
====

.What is @Scope?
[%collapsible]
====
Is an annotation used to define the lifecycle and visibility of a bean. By default Spring treats every bean as a Singleton, meaning it creates only one instace for the entire application. `@Scope` allows you to change that behavior if you need multiple instances or special handling for different user sessions

Spring Bean Scopes

Spring provides several scopes, but the first two are the most common in standard applications:

[cols="1,3", options="header"]

|===
| Scope | Description

| *Singleton* (Default)

| Only one instance is created per Spring IoC container. Everyone who asks for this bean gets the exact same object.

| *Prototype*
| A new instance is created every single time the bean is requested from the container.


| *Request*
| (Web only) A new instance is created for every single HTTP request. The bean dies when the request is finished.

| *Session*

| (Web only) One instance is created per user session. The bean lives as long as the user stays logged in or the session is active.

| *Application*
| (Web only) One instance is created for the lifecycle of the ServletContext.
|===

.What is @Configuration?
[%collapsible]
====
Is a class-level annotation that indicates that a class is a source of bean definitions. It acts a modern alternative to the old XML configuration files.

While annotations like @Service or @Component are great for classes you've written yourself, you often need to use third-party libraries (like RabbitMQ, Security, or Database drivers). .Describe TDD.
[%collapsible]
====
Is a software development process where you write the test before the actual code. The process follows a cycle called Red-Green-Refactor loop

The Red-Green-Refactor Cycle

This three-step process is the heartbeat of TDD:

🔴 RED (Write a failing test): You write a small test for a specific piece of functionality that doesn't exist yet. When you run the test, it must fail (usually because the class or method hasn't even been created).

🟢 GREEN (Make it pass): You write the absolute minimum amount of code required to make that test pass. At this stage, it doesn't matter if the code is "ugly" or inefficient; the only goal is to see a green checkmark.

🔵 REFACTOR (Clean it up): Now that you have a safety net (the passing test), you clean up the code. You remove duplication, improve naming, and optimize performance. You run the test again to ensure your "cleaning" didn't break the logic.


Reduced Debugging: Since you are writing tests in tiny increments, if a test fails, you know exactly which 3 lines of code caused the problem.

Living Documentation: Your tests describe exactly what the system is supposed to do. A new developer can read the tests to understand the business logic.

Simplified Design: TDD naturally pushes you toward Loose Coupling and Dependency Injection, because tightly coupled code is very difficult to test in small units.

Confidence to Change: When you have a massive suite of TDD-created tests, you can refactor or upgrade libraries (like moving from Spring Boot 2 to 3) with the confidence that you'll catch any regressions immediately

.Difference between Set and Map?
[%collapsible]
====
A Set: Is a collection of individual, unique items

A Map: is a collection of key-value pairs used for lookups

Common Implementations

Both structures come in different "flavors" depending on whether you care about the order of the items:

Hash (HashSet / HashMap): The fastest version. It uses a hash table, so the order of items is completely random.

Tree (TreeSet / TreeMap): Keeps the items in their "natural order" (e.g., alphabetical or numerical), but is slightly slower.

Linked (LinkedHashSet / LinkedHashMap): Remembers the order in which you inserted the items.
====

.Difference between HashSet and HashMap
[%collapsible]
====
The Core Structure

HashMap: Implements the Map interface. It stores data in pairs (Key→Value). You use the key to find the value.

HashSet: Implements the Set interface. It stores only individual objects. Its main job is to ensure no duplicates exist in your collection.

.Difference between ArrayList and LinkedList
[%collapsible]
====
1. ArrayList (The Dynamic Array)
+
An ArrayList uses a standard array to store elements. When you create one, it reserves a contiguous block of memory.
+
Accessing Data: Since it uses an array, it knows exactly where each element is located by its index. This makes looking up an item (O(1)) incredibly fast.
+
Resizing: If you fill it up, the ArrayList has to create a new, larger array and copy all the old elements over.
+
Shifting: If you delete an item from the middle, every item after it must slide over to fill the gap, which is slow (O(n)).

2. LinkedList (The Chain)
+
A LinkedList stores elements as Nodes. Each node contains the data and a "pointer" (a reference) to the next and previous nodes in the list.
+
Accessing Data: It doesn't know where the 50th item is. It has to start at the first node and "walk" the chain until it reaches the 50th (O(n)), which is slow.
+
Insertion/Deletion: If you want to add an item to the middle, you don't have to move anything. You simply break two "links" in the chain and point them to your new node. This is very fast (O(1)) once you are at the correct location.
+
Memory: It uses more memory than an ArrayList because it has to store the addresses of the neighboring nodes for every single element.
