.What are the advantages and disadvantages of an object-oriented programming approach?
[%collapsible]
====
Advantages of OOP
The primary benefit of OOP is code reuse, as all the fundamental principles work together to facilitate the efficient use of existing code. Specific advantages include:
* Implementation Hiding (Encapsulation): By wrapping data and methods into a single unit, OOP protects an object's internal state from outside interference.
* Extensibility (Inheritance): It allows for the creation of new entities based on existing ones, enabling developers to build upon existing logic without starting from scratch.
* Flexibility (Polymorphism): It provides the ability to have different forms for the same entity, allowing for more generic and flexible code.
* Effective Modeling (Abstraction): OOP allows for describing complex entities through a set of common characteristics, defining a clear "contract" for how to interact with an object without needing to understand its internal complexity.

Trade offs
* Performance Overhead: Abstract classes and interfaces are slightly slower than concrete implementations because the system must perform a search to locate the correct overridden method at runtime.
* Slower Memory Access: Because OOP relies heavily on objects, data is primarily stored in the Heap, which has slower access, allocation, and deallocation speeds compared to the Stack.
* Thread-Safety Complexity: Unlike Stack memory (which is private to a thread), the Heap is shared among all threads. This means objects are not thread-safe by default and require explicit synchronization, which can increase the complexity of the code and potentially slow down the application.
* Memory Fragmentation: Dynamic allocation on the heap can lead to fragmentation, which requires a Garbage Collection cycle to manage. This process can cause "stop-the-world" events where the application temporarily pauses
====

.What are the basic principles of OOP?
[%collapsible]
====
* `Encapsulation` is the practice of bundling data (variables) and methods that act on that data into a single unit (a class). We use access modifiers to achieve data hiding, ensuring that the interal state of an object is protected and only accessible through public methods like getters and setters.
* `Inheritance` is the creation of a new entity based on an existing one.
* `Polymorphism` is the ability to have different behaviors for different forms of the same entity. Is the ability of a subclass to provide a specific implementation of a method that is already defined in its parent class
* `Abstraction` is the process of hiding the internal complexity and only exposing the essential features  interacting with an entity (often called a contract). This allows developers to interact with a clean, high-level interface without needing to understant the underlaying code machinery.
* `Reuse` - everything listed above works for code reuse.

IMPORTANT: This is the only correct order of OOP paradigms, since each subsequent one uses the previous ones.

.Tell us about the basic concepts of OOP: `class`, `object`, `interface`?
[%collapsible]
====
* A `class` is a way of describing an entity, defining the state and behavior that depends on this state, as well as the rules for interacting with this entity (contract). +
From a programming point of view, a class can be viewed as a set of data (attributes, class members) and functions to work with them (methods). +
From the point of view of the structure of the program, a class is a complex data type.

* An `object` (instance) is a separate representative of a class with a specific state and behavior that is completely determined by the class. Each object has specific attribute values and methods that operate on those values based on the rules defined in the class.

* An `interface` is a collection of class methods available for use. The interface of a class will be a set of all its public methods together with a set of public attributes. Basically, an interface specifies a class, clearly defining all possible actions on it.

In summary, the class defines the structure, the object is the actual entity living in memory, and the interface defines the rules for how external code can interact with that entity

.What are `static` and `dynamic` linking?
[%collapsible]
====
* Attaching a method call to the body of a method is called `binding`. If the binding is done by the compiler (linker) before starting the program, then it is called `static` or `early binding`.

* In turn, `late` binding is binding carried out directly at runtime, depending on the type of object. Late binding is also called `dynamic` or `runtime` binding. In languages that implement late binding, there must be a mechanism for determining the actual type of the object at runtime to invoke the appropriate method. In other words, the compiler does not know the type of the object, but the method invocation mechanism determines it and calls the corresponding method body. `Late-binding` is *language-specific*, but it's easy to assume that some additional information must be included in objects to implement it.

* All Java methods use late (dynamic) binding, unless the method has been declared `final` (private methods are final by default).

* An interface defines a contract (what a class can do); an abstract class defines an identity (what a class is). Use an interface when unrelated classes could share behavior (Serializable, Comparable). Use an abstract class when classes share state or a common base implementation (HttpServlet, AbstractList).

.What are the characteristics of an object?
[%collapsible]
====
* State and Behavior: An object's state is defined by its specific attribute values, while its behavior consists of the methods that operate on those values based on the rules established in its class.
* Memory Location: When an object is created (typically using the new operator), its actual space is allocated in the Heap memory. While the object itself lives in the Heap, a reference to that space is stored in the Stack.

.What are the different ways to create objects?

Objects can be created in the following ways:

1. Using the `new` keyword with a constructor
2. Using the `newInstance()` method (reflection)
3. Using the `clone()` method on an existing object
4. Using deserialization
5. Using factory methods or builders

.How to create an object using the newInstance method?
[%collapsible]
====
The `newInstance()` method is a reflective approach that creates an instance of a class using `Class.forName()` and `getDeclaredConstructor()`. This method requires the class to have a no-argument constructor and throws checked exceptions like `InstantiationException` and `IllegalAccessException`.
====

.How to create a copy of an existing object?
[%collapsible]
====
You can create a copy of an existing object through:

1. Shallow copy: Creating a new object and copying all primitive values and references (shallow copy copies references)
2. Deep copy: Creating a new object with completely independent copies of all objects it references
3. Using the `clone()` method if the class implements `Cloneable`
4. Using copy constructors
5. Using serialization and deserialization
====

.How to use the clone method to create a new object?
[%collapsible]
====
The `clone()` method creates a shallow copy of an object. To use it:

1. The class must implement the `Cloneable` interface (marker interface)
2. Override the `clone()` method (inherited from `Object`)
3. Call `super.clone()` within your override
4. Handle the `CloneNotSupportedException` exception

Note: `clone()` performs a shallow copy by default, so you may need to manually handle deep copying of mutable objects.
====

.When does your code throw the CloneNotSupportedException?
[%collapsible]
====
`CloneNotSupportedException` is thrown when:

1. A class does not implement the `Cloneable` interface but attempts to call `clone()`
2. The `clone()` method explicitly throws the exception
3. A class overrides `clone()` and throws the exception under certain conditions

This exception is a checked exception, so it must be caught or declared in the method signature.
====

.How to serialize an object?
[%collapsible]
====
To serialize an object:

1. Implement the `Serializable` interface (marker interface)
2. Use `ObjectOutputStream` to write the object to a stream
3. All referenced objects must also be serializable

Example:
[source,java]
----
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("file.dat"));
oos.writeObject(myObject);
oos.close();
----
====

.What is `de-serialization`, and how does it generate a new object?
[%collapsible]
====
De-serialization is the process of reconstructing an object from its serialized (binary) form. It:

1. Reads the serialized bytes from a stream using `ObjectInputStream`
2. Reconstructs the object in memory with the same state as when it was serialized
3. Creates a new object instance without calling constructors or initializers (the object state is directly restored from the stream)

Example:
[source,java]
----
ObjectInputStream ois = new ObjectInputStream(new FileInputStream("file.dat"));
Object myObject = ois.readObject();
ois.close();
----

.`default` vs `static`:
[%collapsible]
====
default is about giving implementing classes a ready-to-use or overridable behavior, while static is about utility logic tied to the interface itself, not to any instance.

[source,java]
----
interface Greeter {
    static void staticHello() {
        System.out.println("Hello from static");
    }

    default void defaultHello() {
        System.out.println("Hello from default");
    }
}

class MyClass implements Greeter { }

// static — called on the interface directly
Greeter.staticHello();

// default — called on the instance, inherited
MyClass obj = new MyClass();
obj.defaultHello();

// ❌ this doesn't work — static is NOT inherited
obj.staticHello();
----

.Can interfaces have constructors?

.What is polymorphic reference?
[%collapsible]
====
It is a reference variable that can point to objects of different types at different times, specifically those that share an inheritance or implementation relationship.

**How it is achieved:**

* Supertype reference, subtype object: You achieve a polymorphic reference by declaring a variable of a supertype and assigning it an object of a subtype.

Example:
[source,java]
----
Animal animal = new Dog(); // Dog is a subtype of Animal
animal = new Cat(); // Cat is also a subtype of Animal
----

It allows an application to evolve more easily.

This is why it is recommended to write code against a collection interface (like `List`) rather than a concrete implementation (like `ArrayList`). It allows you to swap the underlying implementation later without breaking the rest of the code.

.If Object is a cosmic superclass, what is wrong if you refer to every object by it?
[%collapsible]

If you refer to an object as a generic `Object`, you can only access the methods defined in the `java.lang.Object` class, such as `toString()`, `equals()`, and `hashCode()`. You lose access to the unique methods and attributes that define the actual entity you are working with.

* Broken "contract": A class is a contract that stablishes the rules for interacting with an entity. By using me most general reference, you are essentially ignoring that contract

In summary, referring to every object as its cosmic superclass discards the specific description and rules, that make that object useful in your application

.If your class implements two interfaces and both interfaces declare similar methods, which method will be invoked? Is it permitted?
[%collapsible]
====
If a class implements two interfaces with similar method signatures:

1. If both interfaces have abstract methods, the implementing class must provide a single implementation that satisfies both contracts
2. If both interfaces have default methods with the same signature (Java 8+), it is a conflict and the class must override the method to resolve the ambiguity
3. The method body of the class's implementation will be invoked (the actual implementation in the class)

This is the solution to the "diamond problem" in Java.

.What is the use of `static` keyword?

.Explain the use of static import
[%collapsible]

Overusing static imports in production code can hurt readability. It's best reserved for cases where the class name is very well known or the repetition is genuinely noisy (like in tests).

[source,java]
----
// Without static import

// With static import

import static java.lang.Math.PI;

PI;       // no need to write Math.PI
----

[%collapsible]

The `super` keyword must be located on the first line of the constructor to ensure that the parent entity is fully initialized before the child class begins with its own initialization.

For a subclass to be a valid representative of that entity, the "existing" parent part of the object must be established first
====

.Do we have destructors in Java?
[%collapsible]
====
No, Java does not have explicit destructors like C++. Instead, Java uses garbage collection to automatically reclaim memory from objects that are no longer in use. The `finalize()` method (deprecated since Java 9) was the closest equivalent, but it is unreliable and should not be used.

Modern alternative: Use try-with-resources or override `close()` for proper resource cleanup.
====

.Which methods in the Object class can be overridden?
[%collapsible]
====
The following methods from the `Object` class can be overridden:

1. `toString()`: Returns a string representation of the object
2. `equals(Object o)`: Defines logical equality
3. `hashCode()`: Returns a hash code for the object
4. `clone()`: Creates a copy of the object
5. `finalize()`: Cleanup before garbage collection (deprecated, should not override)

Methods that **cannot** be overridden:

1. `getClass()`: Returns the runtime class
2. `notify()`, `notifyAll()`, `wait()`: Related to thread synchronization (used but not overridden)
====

.What is the `equals` method, and how is it used?
[%collapsible]
====
It is used to define the logical equality between two objects.

It determines if two separate representatives of a class possess the same "state" or attribute values as defined by the class contract
====

.Which class is a cosmic superclass in Java?
[%collapsible]
====
`java.lang.Object` is the cosmic (universal) superclass in Java. Every class in Java, whether explicitly declared or not, inherits from the `Object` class. It is the root of the class hierarchy and provides fundamental methods like `toString()`, `equals()`, `hashCode()`, `clone()`, `getClass()`, `wait()`, `notify()`, and `notifyAll()`.
====

.What is the difference between an equal operator (= =) and equals method?
[%collapsible]
====
* Equal operator (==): This operator compares references. Because object references are stored in the stack while the actual object lives in the heap, using `==` checks if two reference variables point to the exact same memory address

* `equals()` Method: This compares the actual content or state of the objects

[source,java]
----
int a = 5;
int b = 5;

a == b; // ✅ true — comparing the actual values directly
        // no heap involved, no addresses, just 5 == 5      // no need to write Math.PI

// the tricky case

String a = "hello";
String b = "hello";
a == b; // ✅ true — but only because Java reuses the same object
        // from the String pool, so the address happens to be the same

String c = new String("hello");
String d = new String("hello");
c == d; // ❌ false — two explicitly new objects, different addresses
c.equals(d); // ✅ true — same content
----
====

.What is a var argument? Where can it be used?
[%collapsible]
====
Var arguments (varargs) allow a method to accept a variable number of arguments of the same type. Declared using ellipsis (`...`), varargs are converted into an array internally.

Syntax:
[source,java]
----
public void myMethod(String... args) {
    // args is treated as String[]
}
----

Varargs can be used in:
1. Any method parameter position (but only as the last parameter)
2. Constructors
3. Custom methods (any method signature)

Multiple varargs in one method are not allowed.
====

.What are the different rules for declaring the var argument method?
[%collapsible]
====
Rules for declaring varargs:

1. Varargs must be the last parameter in a method signature
2. Only one varargs parameter is allowed per method
3. The varargs parameter is treated as an array internally
4. You can have other parameters before the varargs parameter
5. When calling a method with varargs, you can pass 0 or more arguments
====

.What is the difference between using the var argument and array as a method argument?
[%collapsible]
====
|===
| Varargs | Array

| Accepts 0 or more arguments | Must pass an array object
| More convenient syntax | Requires creating an array explicitly
| Automatically converted to array | Array is passed directly
| Can be mixed with other parameters | Typically the last approach for flexibility
|===

Example:
[source,java]
----
// Varargs - convenient
method("arg1", "arg2", "arg3");

// Array - requires explicit array creation
method(new String[]{"arg1", "arg2", "arg3"});
----
====

.What do you mean by method overloading?
[%collapsible]
====
Method overloading is a mechanism that allows a class to have multiple methods with the same name but different parameter lists. The compiler determines which method to call based on the number, type, and order of arguments at compile time (static binding).

Methods must differ in:
1. Number of parameters
2. Type of parameters
3. Order of parameters

Methods cannot overload based on return type alone.

Example:
[source,java]
----
public void print(int a) { }
public void print(String s) { }
public void print(int a, String s) { }
----
====

.Can we overload a static method?
[%collapsible]
====
Yes, static methods can be overloaded just like instance methods. However, you cannot override a static method. If a subclass defines a static method with the same signature as the parent, it is called method hiding, not overriding.

Example:
[source,java]
----
public static void method(int a) { }
public static void method(String s) { }
----
====

.What is method overriding?
[%collapsible]
====
Method overriding is the ability of a subclass to provide a specific implementation of a method that is already defined in its parent class. The method signature (name, parameters, return type) must be the same. Overriding uses dynamic binding (late binding), meaning the actual method called is determined at runtime based on the object's type.
====

.What are the rules of method overriding in Java?
[%collapsible]
====
Rules for method overriding:

1. The method name and parameter list must be exactly the same
2. The return type must be the same or a covariant type (subtype)
3. The access modifier cannot be more restrictive (can be the same or broader)
4. Cannot throw a checked exception that is broader than the parent method
5. The method cannot be `final` in the parent class
6. The method cannot be `static` (static methods cannot be overridden, only hidden)
7. Must be an instance method (not static)
====

.Can we override a static method?
[%collapsible]
====
No, static methods cannot be overridden. However, static methods can be hidden. If a subclass defines a static method with the same signature as the parent, the subclass version hides the parent version. The method called depends on the reference type (compile-time type), not the object type.
====

.What is a co-variant return type?
[%collapsible]
====
A covariant return type allows an overriding method to return a subtype of the return type declared in the parent class method. This is allowed in Java 5 and later.

Example:
[source,java]
----
class Animal { }
class Dog extends Animal { }

class Parent {
    public Animal getAnimal() { return new Animal(); }
}

class Child extends Parent {
    @Override
    public Dog getAnimal() { return new Dog(); }
}
----

In this example, `Dog` is a covariant return type of `Animal`.
====

.Can we change the return type of the overridden method?
[%collapsible]
====
The return type of an overridden method can only be changed to a covariant type (a subtype of the original return type). Changing the return type to an unrelated type will result in a compilation error.

If the original method returns `Object`, the overriding method can return `String`, `Integer`, or any subclass. However, you cannot change `Object` to a primitive type or an unrelated class.
====

.What is a `marker interface`?
[%collapsible]
====
A marker interface is an interface with no methods or properties. It is used to mark or tag classes that implement it with a specific attribute or capability. Marker interfaces are used to signal to the compiler or runtime that a class possesses a certain property.

Examples:
1. `Serializable`: Marks objects that can be serialized
2. `Cloneable`: Marks objects that can be cloned

Marker interfaces are being replaced by annotations in modern Java.

[source,java]
----
import java.io.*;

class Person implements Serializable { // just tagging it, no methods to implement
    String name;
    Person(String name) { this.name = name; }
}

Person p = new Person("Carlos");
ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("person.dat"));
out.writeObject(p); // ✅ JVM sees the tag and allows it
----
====

.What is the use of a marker interface?
[%collapsible]
====
Marker interfaces are used to:

1. Tag or categorize classes at runtime
2. Provide type-checking by implementing an interface
3. Signal to frameworks (like serialization) that a class supports certain operations
4. Enable compile-time and runtime checks for specific capabilities

Modern practice often replaces marker interfaces with annotations for better flexibility and readability.
====

.What is typecasting in Java, and which are the different types of typecasting available in Java?
[%collapsible]
====
Typecasting is the process of converting a variable from one type to another.

**Types of typecasting:**

1. **Widening (Implicit) Casting**: Automatic conversion from a smaller to a larger type (e.g., `int` to `long`, `int` to `double`). No data loss occurs.
   [source,java]
   ----
   int a = 5;
   long b = a; // Implicit conversion
   ----

2. **Narrowing (Explicit) Casting**: Manual conversion from a larger to a smaller type. Risk of data loss.
   [source,java]
   ----
   long a = 100;
   int b = (int) a; // Explicit conversion
   ----

3. **Reference Typecasting**: Converting between reference types in an inheritance hierarchy.
   [source,java]
   ----
   Animal animal = new Dog(); // Upcasting (implicit)
   Dog dog = (Dog) animal; // Downcasting (explicit)
   ----
====

.What are the types of variables in Java? Also explain their scope.
[%collapsible]
====
* Local Variables: These are variables declared within a method, including parameters and object references. They are stored in the Stack memory. Their scope is highly restricted; they are visible only to the owner thread, and the memory they occupy is automatically reclaimed as soon as the thread or method finishes its execution.

* Instance Variables: These variables define the state and attributes of a separate representative (object) of a class. When an object is created using the new operator, instance variables are allocated space in the Heap memory. Their scope is broader than local variables, as the Heap is visible to all threads and shared among them.

* Static Variables: Also known as class variables, these are shared by all objects of a class. They are loaded into a specialized memory area called Metaspace (or Method Area) only once, specifically when the class is first loaded by the classloader.

* Reference Variables: These are a specific type of variable that stores the address (reference) of an object's location in the Heap rather than the object's data itself. While the object lives in the Heap, the reference variable typically resides in the Stack.
====

.What is an instance initializer block in Java?
[%collapsible]
====
Is a block of code used to initialize the instace variables of a class. It is understood as a tool for defining the initial state of an object alongside the constructor

An instance initializer block is a block of code written directly inside a class body (not inside any method or constructor) that runs every time an object is created, before the constructor body executes. It is useful for sharing initialization logic across multiple constructors without code duplication.
====

.What is the difference between an initializer block and a constructor?
[%collapsible]
====
You use an initializer block for logic that must run for every instance (or the class itself) without exception, while you use a constructor to provide flexible, parameterized initialization for specific objects.
====

.What is difference between a static initializer block and instance initializer block in Java?
[%collapsible]
====
* Static initializer block: Is called only once during the entire application lifecycle, specifically when the class is loaded into memory.

Use a static initializer block when:
You need to initialize a static variable with logic that's too complex for a single line.

* Instance initializer block: (much like a constructor) It is associated with the creation of individual objects

Use an instance initializer block when:
You have initialization logic that needs to run for every object and is shared across multiple constructors, avoiding repetition.
====

.Can you explain the concept of wrapper classes in Java?
[%collapsible]
====
It refers to classes that encapsulate or "wrap" an entity to provide additional functionality, or allow that entity to be used in contexts where its original form is not supported

** Auto-boxing and Auto-unboxing: Wrapper classes are the foundation for auto-boxing and auto-unboxing. These processes describe the automatic conversion between primitive types and their corresponding wrapper objects, which is a common scenario in Java development.

** Use in Collections: A primary use case for these classes is enabling the use of collections with primitive types. Since the collection framework is designed to work with objects, wrapper classes allow primitive data to be stored and manipulated within collection structures like Lists or Sets.

** Servlet Request and Response Wrappers: In the context of web applications, specific wrapper classes like ServletRequestWrapper (including HttpServletRequestWrapper) and ServletResponseWrapper exist. These allow developers to implement custom handlers by adding new methods or overriding existing ones to modify the behavior of standard request and response objects.
====


.What is auto-boxing and auto un-boxing? In which scenario are they observed?
[%collapsible]
====
* Auto-boxing is the automatic conversion that the Java compiler makes between primitive types (such as int, double, or boolean) and their corresponding wrapper objects (such as Integer, Double, or Boolean).
* Auto-unboxing is the reverse process, where a wrapper object is automatically converted back into its original primitive type. These mechanisms are built upon wrapper classes, which encapsulate primitives to provide object-oriented functionality, manage state, and allow them to be used where objects are required.
====

.What is the use of an immutable object? Why we should create such an object?
[%collapsible]
====
The primary use is to ensure that its state cannot be changed once it has been created. There are several critical reasons to create them:

* Thread safety: Since their state never changes, multiple threads can access them simultaneously without the risk of unpredictable results or the need for complex synchronization.

* Collection integrity: The sources highlight that keys used for `Maps` and values used for `Sets` should ideally be of an immutable type. This ensures that once an object is placed in a hash-based collection, its hashcode remains constant, preventing the object from being "lost" within the collection.
====

.What is an enum to you?
[%collapsible]
====
An enum is a special "class" that represents a group of constants (unchangeable variables).
====

.Can enum extend any other class?
[%collapsible]
====
No. All enums in Java implicitly extend `java.lang.Enum`. Since Java does not support multiple inheritance for classes, an enum cannot extend any other class.
====

.Can we write an enum that implements an interface?
[%collapsible]
====
Yes. Enums can implement one or more interfaces, allowing them to provide specific implementations for interface methods.
====

.Can we declare constructors within enum?
[%collapsible]
====
Yes. Enums can have constructors to initialize the state of each enum constant.
====

.Why the constructor of enum is kept as private?
[%collapsible]
====
To ensure that the enum cannot be instantiated from outside. This guarantees that only constants defined within the enum itself ever exist.
====

.Can we instantiate enum using the new keyword?
[%collapsible]
====
No. Enums cannot be instantiated with `new`. The JVM handles the creation of enum instances when the enum is first referenced.
====

.Explain some important methods of enum in Java
[%collapsible]
====
* `values()`: Return an array containing all the constants in the enum
* `valueOf(String)`: Returns the enum constant with the specified name
* `ordinal()`: Returns the position (index) of the constant in the enum declaration
* `name()`: Returns the exact name of the constant as a String
====

.Which package is available as a default package for every Java class?
[%collapsible]

The `java.lang` package is available as a default package for every Java class. It is automatically imported by the JVM, allowing you to use common classes like `String`, `Object`, `Integer`, and others without explicitly importing them.
